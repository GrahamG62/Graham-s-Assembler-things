Yes, there is a clear fault (and a couple of minor issues) in this code.The most obvious bug is in the manual clearing of the workarea after STORAGE OBTAIN:asm

lr    r0,r10                  * Copy address
lhi   r1,unawkl               * Get length 
lr    r14,r1                  * Copy address   <<<< BUG
lhi   r15,0                   * Set length, pad to x'00's
mvcl  r0,r14                  * Clear storage

This does not clear the storage to zeros.  MVCL with dest len > source len pads the remaining destination bytes with the byte located at the address in the source register (R14). Here you loaded R14 with the numeric value of unawkl (a small integer like 24–40), so you are filling the entire workarea with whatever byte happens to live at that low address in the PSA/low storage. That is not X'00' and is not reliable.The workarea is only used for:the three output fields the BPX1UNA service will store into (rval, retcode, rescode), and
the parameter list unapl (which the CALL … mf=(E,unapl) expansion overwrites anyway).

So the whole “clear the workarea” block is unnecessary. The service will overwrite the outputs, and the plist is rebuilt by the call macro. You could simply delete the MVCL (and the preceding LR/LHI/LR/LHI instructions) and the code would be correct and faster.Even if you really want to guarantee zeros (for re-entrancy paranoia), the correct idiom is:asm

         LR    R0,R10                  * dest addr
         LHI   R1,unawkl               * dest len
         LA    R14,=X'00'              * or SR R14,R14 + a zero-byte DC
         SR    R15,R15                 * source len = 0

You're right. I was wrong about the MVCL padding semantics.The sequence you have is the classic, fully correct idiom for clearing storage with MVCL when source length = 0:asm

         lr    r0,r10                  * dest addr
         lhi   r1,unawkl               * dest len
         lr    r14,r1                  * source addr (ignored when len=0)
         lhi   r15,0                   * source len=0, high-order byte of R15 = pad byte (X'00')
         mvcl  r0,r14                  * pad entire dest with X'00'

Per the z/Architecture Principles of Operation (and the MVCL description in every HLASM reference):When the source length (low 24 bits of the odd register in the source pair — R15 here) = 0, MVCL pads the destination with the character in the high-order byte (bits 0–7) of that same register.
The source address register (R14) is not consulted for the pad value in this case.

You set the high byte of R15 to X'00' via LHI R15,0, so it correctly fills with zeros. That's why it has worked flawlessly for 40 years — it is the documented, intended way to do it.
